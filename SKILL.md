# TDX API Usage

按本 skill 对接 TDX A 股股票数据 API。目标是稳定地拿到数据、解释字段、校验数据边界，并输出有利于后续分析与决策的数据支持结果。

## 何时使用

- 用户要查询或调试 `quote`、`kline`、`minute`、`trade`、`trade-history`、`search`、`stock-info`、`codes`、`index`、`workday`、`income` 等股票 HTTP API。
- 用户要生成 Python、curl、JavaScript 等调用示例。
- 用户要核对接口参数、响应结构、单位换算或错误处理方式。
- 用户希望让 AI 基于更准确的股票数据做后续分析、比较、归纳或决策。

## 角色定位

- 本 skill 是面向上层 AI 的股票数据支持层，不替代上层 AI 的综合判断。
- 本 skill 应全力为 AI 提供准确、可解释、可比较、可用于后续分析与决策的股票数据结果。
- 本 skill 不应只返回未经整理的原始接口内容；应尽可能把关键字段、单位、时间范围、数据状态和异常信息整理清楚，降低上层 AI 的二次处理成本。

## 不适用场景

- 不用于基金净值、基金档案、债券、期货、宏观经济、外汇、数字货币等非股票场景。
- 当用户只给出六码纯数字代码时，不要默认它一定是股票代码；例如 `001309` 既可能被用户当作基金代码，也会被本 API 解析为股票 `德明利`。
- 如果用户明确要查基金，应该改用基金数据源，而不是继续调用本 skill 中的股票接口。

## 硬性约束

- host 固定为 `tdx.acdzh.xyz`。
- 若用户没有额外指定协议或端口，示例默认使用 `http://tdx.acdzh.xyz` 作为 base URL。
- 命令行脚本或 Python 客户端访问业务接口时，建议显式携带 `User-Agent` 请求头；实测 `/api/search`、`/api/quote`、`/api/kline` 等接口在缺少常见浏览器式请求头时可能返回 `403`。
- 不要把一次性抓数脚本写进仓库。
- 如果需要写一次性的 Python 脚本，必须先生成 UUID，并把脚本写入系统临时目录，文件名使用该 UUID。
- `quote`、`kline`、`minute`、`trade`、`trade-history`、`index` 中的大多数价格字段默认单位为 `厘`，展示给用户时通常需要换算为 `元`，即 `value / 1000`。
- `income` 接口返回的价格字段实测已经是 `元`，不要再次除以 `1000`。
- 返回成交量默认单位通常为 `手`，若需要换算成 `股`，使用 `value * 100`。

## 推荐流程

1. 明确用户要的接口、资产类型、股票代码、日期、周期和返回格式。
2. 若用户只给出六码纯数字代码，先确认这是股票还是基金；未确认前不要直接调用股票接口。
3. 若股票代码不明确，优先调用 `/api/search`。
4. 若需要检查服务可用性，先调用 `/api/server-status`。
5. 组装请求 URL 或请求体，优先给出最小可执行示例，并在示例里带上 `User-Agent`。
6. 读取统一响应结构中的 `code`、`message`、`data`。
7. 对价格、成交量、成交额做必要单位换算，并在结果里显式说明具体接口的单位前提。
8. 在输出前先整理关键字段、时间范围、数据状态和异常信息，避免把未经解释的原始字段直接抛给上层 AI。
9. 若 `code != 0` 或列表为空，明确区分“接口报错”和“该日期无数据”。

## 一次性 Python 脚本规则

只有在以下情况才建议写一次性 Python 脚本：

- 需要连续调用多个接口。
- 需要对响应做筛选、排序、聚合或换算。
- 需要比 curl 更稳定的超时、异常处理和 JSON 输出。

脚本创建流程必须严格遵循下面的顺序：

1. 先生成 UUID。
2. 以该 UUID 作为文件名，写入系统临时目录。
3. 执行脚本。
4. 用完后可删除该临时脚本。

参考命令：

```bash
SCRIPT_UUID="$(python - <<'PY'
import uuid
print(uuid.uuid4())
PY
)"
SCRIPT_PATH="/tmp/${SCRIPT_UUID}.py"
echo "$SCRIPT_PATH"
```

Python 示例模板：

```python
import requests

BASE_URL = "http://tdx.acdzh.xyz"
HEADERS = {"User-Agent": "Mozilla/5.0"}

resp = requests.get(
    f"{BASE_URL}/api/quote",
    params={"code": "000001"},
    headers=HEADERS,
    timeout=10,
)
resp.raise_for_status()
payload = resp.json()

if payload["code"] != 0:
    raise RuntimeError(payload["message"])

quote = payload["data"][0]
print({
    "code": quote["Code"],
    "close_yuan": quote["K"]["Close"] / 1000,
})
```

## 统一响应处理

所有接口都按下面结构处理：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

处理规则：

- `code == 0` 视为成功。
- `code != 0` 时，优先向用户透出 `message`。
- 只有在 `code == 0` 的前提下再解析 `data`。
- 业务接口可能返回 HTTP `403`；若出现这种情况，先检查是否缺少 `User-Agent` 等常见请求头，而不是立即认定服务不可用。

## 常用接口选择建议

- 实时五档行情：`GET /api/quote`
- K 线：`GET /api/kline`
- 分时：`GET /api/minute`
- 逐笔成交：`GET /api/trade`
- 搜索股票：`GET /api/search`
- 股票概览：`GET /api/stock-info`
- 股票列表：`GET /api/codes`
- 批量行情：`POST /api/batch-quote`
- 指数：`GET /api/index`
- 交易日：`GET /api/workday`
- 收益区间：`GET /api/income`

当数据源必须明确区分时，优先使用：

- 通达信原始全量 K 线：`GET /api/kline-all/tdx`
- 同花顺前复权全量 K 线：`GET /api/kline-all/ths`

## 易错点

- `/api/minute` 和 `/api/trade` 在指定日期无数据时，实测返回 `{"Count":0,"List":null}`；这不等于接口失败。
- `/api/trade-history` 单次最多返回 2000 条，需要通过 `start` 和 `count` 分页。
- `/api/kline-all` 数据量可能很大，优先带 `limit`。
- 当用户只给股票名称时，不要直接猜代码，先调用 `/api/search`。
- 当用户只给六码纯数字代码时，不要默认它一定是股票；先确认资产类型，避免把基金代码误判成股票代码。
- 当用户关心“前复权还是原始数据”时，不要只用 `/api/kline`，要改用显式数据源接口。
- `/api/income` 的价格字段实测已经是 `元`，不要沿用 `quote`/`kline` 的 `厘 -> 元` 规则。

## 输出要求

- 默认优先输出可供后续分析与决策直接使用的数据摘要，而不是大段原始 JSON。
- 结果中优先说明查询范围、关键数据、单位换算、时间范围、数据状态和异常提示。
- 当字段对后续分析有帮助时，要把原始字段翻译成易理解的数据含义，例如价格、涨跌、成交量、区间表现、是否复权、是否盘中。
- 仅在用户明确需要调试或实现接口调用时，再展开请求方法、路径、参数和示例响应关键字段。
- 如果用了临时 Python 脚本，明确说明脚本位于临时目录，且文件名来自 UUID。

## 标准数据摘要模板

- 默认按“查询范围 -> 关键数据 -> 数据说明 -> 异常与边界 -> 原始接口”这一顺序组织结果，便于上层 AI 直接消费。
- 若用户的目标明显偏分析或比较，优先补齐更有利于后续推理的数据摘要；若用户目标明显偏调试，才把接口细节前置。
- 除非用户明确要求查看完整响应，否则不要直接贴大段原始 JSON。

推荐模板：

```text
查询范围：
- 标的：000001 平安银行
- 接口：quote / kline(day) / minute
- 时间范围：2026-05-26 收盘前后；日线最近 30 个交易日

关键数据：
- 最新价：10.68 元
- 当日开高低收：10.71 / 10.83 / 10.66 / 10.68 元
- 涨跌幅：0.09%
- 成交量：837346 手（约 8373.46 万股）
- 成交额：900879552 元

数据说明：
- 价格字段来自 `quote.K`，原始单位为厘，已换算为元
- 数据为 A 股股票实时/近实时行情
- 若使用 `kline-all/ths`，需显式标注为前复权数据

异常与边界：
- 当前结果未发现接口报错
- 若为盘中数据，数值可能继续变化
- 若只给出六码纯数字代码，已先按股票代码核对标的身份

原始接口：
- `GET /api/quote?code=000001`
- `GET /api/kline?code=000001&type=day`
```

## 常见任务的摘要重点

- 查询单只股票实时情况时，优先整理：最新价、开高低收、涨跌幅、成交量、成交额、买卖盘或量价活跃度。
- 查询日线或区间表现时，优先整理：时间范围、区间涨跌、最高/最低、成交量变化、数据源是否复权。
- 查询分时或逐笔成交时，优先整理：查询日期、是否有数据、成交活跃度、价格波动区间、数据是否完整。
- 查询多只股票对比时，优先整理：各标的核心字段口径一致后的并列结果，避免混用不同单位或不同复权方式。
- 查询交易日或收益区间时，优先整理：起止日期、计算口径、返回字段单位，以及结果是否适合直接与行情接口比较。

## 常用接口摘要示例

### `quote` 示例

```text
查询范围：
- 标的：000001 平安银行
- 接口：GET /api/quote
- 时间：当前实时/近实时行情

关键数据：
- 最新价：10.68 元
- 昨收：10.79 元
- 今开：10.71 元
- 最高/最低：10.83 / 10.66 元
- 涨跌幅：0.09%
- 成交量：837346 手（约 8373.46 万股）
- 成交额：900879552 元

数据说明：
- `quote.K` 中价格原始单位为厘，已换算为元
- 成交量原始单位为手；如需换算为股，乘以 100
- 该结果适合供上层 AI 做实时强弱、量价活跃度、盘中状态判断

异常与边界：
- 若为交易时段内数据，数值可能继续变化
- 若请求返回 403，先补 `User-Agent`
```

### `kline` 示例

```text
查询范围：
- 标的：000001 平安银行
- 接口：GET /api/kline?code=000001&type=day
- 时间范围：最近 30 个交易日日线

关键数据：
- 区间最新收盘价：10.68 元
- 区间最高/最低：11.22 / 10.93 元
- 区间成交量变化：近期放量后回落
- 数据点数量：30

数据说明：
- `kline.List` 中价格原始单位为厘，已换算为元
- 默认是该接口返回口径，不自动等同于前复权数据
- 该结果适合供上层 AI 做区间走势、波动范围、阶段强弱比较

异常与边界：
- 历史极早期数据可能出现异常值，应结合时间范围核对
- 若用户明确关心复权口径，优先改用 `kline-all/ths` 或显式说明当前口径
```

### `minute` 示例

```text
查询范围：
- 标的：000001 平安银行
- 接口：GET /api/minute
- 日期：20241103

关键数据：
- 查询结果：当日无分时数据
- 记录数：0

数据说明：
- 当指定日期非交易日、停牌或源端无记录时，可能无数据
- 该结果适合供上层 AI 判断“该日期是否可做盘中走势分析”

异常与边界：
- 实测空结果结构为 `{"Count":0,"List":null}`，这不等于接口失败
- 不要把无数据误解为服务异常
```

### `income` 示例

```text
查询范围：
- 标的：000001 平安银行
- 接口：GET /api/income
- 起点日期：2024-01-01
- 观察区间：5 日、20 日

关键数据：
- 5 日收益：-0.03 元，收益率约 -0.39%
- 20 日收益：0.29 元，收益率约 3.79%
- 对应终点日期：2024-01-09、2024-01-30

数据说明：
- `income` 中价格字段实测已是元，不需要再除以 1000
- 返回的是基于起点价格的区间收益结果，适合做阶段表现对比
- 该结果适合供上层 AI 做回看收益、区间相对表现、阶段收益比较

异常与边界：
- `income` 的单位口径不同于 `quote` / `kline`，不要混用换算规则
- 比较不同标的时，需确保起始日期和观察区间一致
```

### `search` 示例

```text
查询范围：
- 关键词：平安
- 接口：GET /api/search
- 目标：确认股票代码与交易所

关键数据：
- 命中结果数：3
- 候选标的：601318 中国平安、000001 平安银行、001359 平安电工

数据说明：
- `search` 结果用于确认代码、名称、交易所，不直接等同于行情结果
- 当关键词可能对应多只股票时，应把候选列表整理清楚供上层 AI 继续选择或追问
- 该结果适合供上层 AI 做标的识别、消歧、后续接口取数准备

异常与边界：
- 关键词过短或过泛时，命中结果可能较多
- 纯数字代码可能存在跨资产类别歧义，命中股票结果后仍需核对用户真实意图
```

### `stock-info` 示例

```text
查询范围：
- 标的：000001 平安银行
- 接口：GET /api/stock-info
- 内容：概览信息 + 近期日线 + 分时相关数据

关键数据：
- 返回内容包含：`quote`、`kline_day`、`minute`
- 近期日线数量：30
- 适合一次性获取单只股票的概览型数据

数据说明：
- `stock-info` 是聚合接口，适合先做单股概览，再决定是否继续调用更细分接口
- 应优先整理其中最重要的概览数据，而不是直接展开整个嵌套结构
- 该结果适合供上层 AI 做“先看全貌，再决定下一步取数”的分析准备

异常与边界：
- 聚合接口返回内容较多，直接贴原始结构会增加上层 AI 的处理负担
- 需要明确当前整理出的关键数据来自 `quote`、`kline_day` 还是 `minute`
```

### `batch-quote` 示例

```text
查询范围：
- 标的：000001 平安银行、600519 贵州茅台
- 接口：POST /api/batch-quote
- 目标：多只股票并列对比实时行情

关键数据：
- 000001：最新价 10.68 元，涨跌幅 0.09%，成交量 837346 手
- 600519：最新价 1285.88 元，涨跌幅约 0.98%，成交量 45931 手
- 返回标的数：2

数据说明：
- `batch-quote` 适合做同口径的多标的实时对比
- 应统一换算所有标的的价格和成交量单位后再并列展示
- 该结果适合供上层 AI 做横向比较、相对强弱观察、候选标的筛选

异常与边界：
- 若某只股票未返回，应显式标注缺失标的，而不是默认全部成功
- 多标的对比时要避免混入不同时间点或不同数据源口径
```

### `trade-history` 示例

```text
查询范围：
- 标的：000001 平安银行
- 接口：GET /api/trade-history
- 日期：20241101
- 分页：start=0, count=2

关键数据：
- 返回条数：2
- 成交示例：
  - 14:57，11.44 元，56 手
  - 15:00，11.43 元，11586 手

数据说明：
- `trade-history` 适合查看某日历史逐笔成交明细
- 价格原始单位为厘，已换算为元；成交量原始单位通常为手
- 该结果适合供上层 AI 做尾盘成交观察、活跃度分析、关键时段成交特征整理

异常与边界：
- 单次最多 2000 条，需要通过 `start` 和 `count` 分页
- 逐笔成交数据量可能较大，不宜一次性把完整明细原样输出给上层 AI
```

## 输出禁忌

- 不要只返回字段名和原始数值，不做任何单位和语义整理。
- 不要混合未复权、前复权、实时、历史等不同口径的数据却不说明。
- 不要在未核对标的身份、时间范围或接口状态前，把结果包装成可直接用于后续分析的数据。
- 不要省略异常提示；只要存在空数据、歧义代码、接口降级、HTTP 403 或单位差异，都应显式写出。
